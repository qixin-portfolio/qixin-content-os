# v0.3.0 Phase 3 Release Baseline

日期：2026-07-12
阶段：Phase 3 Content Intelligence Layer 内容智能层

## 阶段目标

在真实、可追溯的 EventCard 上增加内容机会判断和人工确认后的母内容草稿生成：

`SourceItem → EventCard → ContentScore → ContentAngle → MasterContent draft`

本阶段不接入外部模型、向量数据库、RAG、多 Agent、多模型路由，也不自动发布。

## 数据模型

- 新增 `ContentScore`：保存新鲜度、个人性、行业性、画面性、业务性五项 0-20 评分、总分、推荐级别和原因。
- 新增 `ContentAngle`：保存角度类型、标题、核心观点、目标受众、推荐平台、原因和人工选择状态。
- 新增 `VoiceProfile`：保存平台、语气、偏好词、禁用词、写作规则和示例文本 JSON。
- `EventCard` 增加 `contentScore` 和 `contentAngles` 关系。
- `MasterContent` 增加 `factReferencesJson`，保存生成草稿引用的 SourceItem ID。
- 新增 migration：`20260712120000_add_content_intelligence`。

## 评分规则

每个维度限制在 0-20，总分为五项之和：

- 完整事实字段提供基础分。
- 结果、个人反思、问题与结果对照、证据类型、多个 SourceItem、产品过程、个人判断和行业/产品意义分别提供确定性加分。
- `80-100` 推荐 `publish_now`。
- `55-79` 推荐 `combine_later`。
- `0-54` 推荐 `archive_only`。

推荐级别只影响内容建议，不删除事件，也不代表已发布。

## ContentAngle 与 VoiceProfile

- 高分事件最多生成五个事实型角度，至少三个；低分事件最多两个。
- 角度只重新组织已有的 `whatHappened`、`problem`、`result`、`whyItMatters` 和 `personalReflection`，不新增用户、收入、客户、上线或成果数据。
- 生成前由人工选择角度和 VoiceProfile。
- mock 生成器会移除 VoiceProfile 的禁用词，并要求 EventCard 具有 SourceItem 引用。
- `archive_only` 事件不能生成 MasterContent。

## 页面与 API

- `/opportunities`：展示事件、项目、五项评分、总分、推荐级别、评分原因和角度数量。
- `/opportunities/[eventId]`：展示证据来源、评分、角度和 VoiceProfile 选择。
- `GET /api/opportunities`：获取内容机会列表。
- `GET /api/opportunities/[eventId]`：获取单个事件的机会详情。
- `POST /api/opportunities/[eventId]/generate`：人工确认角度和 VoiceProfile 后生成 MasterContent draft；已有母内容返回 `409`，不覆盖。

## 透明工地真实验收

seed 使用已有透明工地 README、版本记录、产品一页纸和人工补充的事件字段。当前可确认的内容是产品文档已形成；截图、后台版本记录、代码路径和真实项目案例仍待补证。

验收结果：

- ContentScore：`74/100`
- 推荐：`combine_later`
- 维度：新鲜度 `20`、个人性 `20`、行业性 `12`、画面性 `12`、业务性 `10`
- ContentAngle：5 个
- MasterContent：1 个 `drafting` 草稿
- fact references：4 个 SourceItem ID

没有写入或推断用户数量、收入、客户反馈、商业成交或正式上线状态。

## 已知限制

- 使用 deterministic mock provider，尚未接入 OpenAI、Claude、DeepSeek 或豆包。
- 尚未生成或持久化四个平台的本阶段适配草稿；平台内容仍需后续人工审核流程。
- 尚未接入向量数据库、RAG、多 Agent、多模型路由、自动发布或发布数据回流。
- 本机 Prisma 7.8 的 `migrate dev`/`migrate deploy` 在 schema engine 阶段返回空错误；migration SQL 已由 `migrate diff` 生成并使用 `db execute` 验证，不能将标准 migration deploy 写成通过。
- seed 依赖本机真实资料根目录，资料缺失时会失败，不生成替代内容。
