## 审查中发现的问题

原实现已具备单事务、源 Revision 唯一键和同进程锁，但最终审查发现两个缺口：

1. 返回体只有批准 Revision 的通用 `id`，没有显式 `approvalRevisionId` 和 `sourceRevisionId`。
2. 两个独立进程同时批准时，一个成功，另一个在 `StyleReview.create` 处返回 Prisma `P1008 Operation has timed out`。原冲突后单次查询发生得过早，未稳定返回第一组批准结果。

本次先用测试复现 RED，再做最小修复：增加稳定 ID 字段；对明确可重试的 SQLite/Prisma 竞争错误做最多 3 次有限重试。

## 幂等键与数据库约束

```prisma
DraftRevision.approvedSourceRevisionId String? @unique
VoiceSample.sourceRevisionId           String? @unique
```

语义：

- `human_approval.approvedSourceRevisionId` 指向被批准的 `human_edit` 或 `ai_suggestion` 源 Revision。
- `approved_draft.sourceRevisionId` 指向同一个源 Revision。
- 两个字段都不指向 `human_approval` Revision 自身。
- 普通 `ai_initial`、`human_edit`、`ai_suggestion` 的 `approvedSourceRevisionId` 为空。
- `imported_post` VoiceSample 的 `sourceRevisionId` 为空。

数据库唯一索引：

- `DraftRevision_approvedSourceRevisionId_key`
- `VoiceSample_sourceRevisionId_key`

SQLite 允许唯一索引中存在多个 NULL。测试在同一临时库创建多个 NULL Revision 和两个 NULL `imported_post` 样本成功；非 NULL 源 ID 重复插入时 Prisma 返回 `P2002`。

同一 EditorialDraft 的不同源 Revision 使用不同非 NULL 幂等键，因此可以分别产生各自的批准 Revision 和 VoiceSample。

## 事务边界

首次批准在一个 Prisma 事务中完成：

1. 读取 EditorialDraft、当前 Revision 和 VoiceProfile。
2. 解析被批准的源 Revision。
3. 按源 Revision 查询既有批准产物。
4. 创建并检查 StyleReview。
5. 创建 `human_approval` Revision。
6. 更新 Draft 的 currentRevisionId、status 和 approvedAt。
7. 创建 `approved_draft` VoiceSample。

任一步失败全部回滚。调用方不需要先查询 Draft status。

## 同进程锁与跨实例并发

模块级锁只按 EditorialDraft 串行化同一 Node.js 进程内的批准请求，用于减少无意义竞争，不是最终正确性来源。

跨实例测试使用：

- 两个独立 Node 进程。
- 两个独立 PrismaClient。
- 各自独立加载 revision service，因此不共享模块级锁。
- 同一启动时间批准同一个临时源 Revision。

首次运行结果：一个成功，另一个返回 `P1008`，测试 RED。

修复后结果：

- 两个调用均返回可识别结果。
- 一个 `idempotent: false`，另一个 `idempotent: true`。
- `approvalRevisionId`、`voiceSampleId`、`sourceRevisionId` 完全一致。
- 最终只有 1 个 `human_approval` Revision 和 1 个 `approved_draft` VoiceSample。
- 没有未处理的 unique constraint 500，没有部分写入。

## Unique Conflict 与有限重试

服务只重试以下竞争错误：

- Prisma `P1008`：操作超时。
- Prisma `P2002`：唯一约束冲突。
- Prisma `P2028`：事务状态错误。
- Prisma `P2034`：事务写冲突或死锁。
- SQLite `database is locked` / `database is busy`。

最多执行 3 次，等待间隔为 50ms、100ms，不无限重试。每次失败后先按源 Revision 查询数据库中的唯一批准产物；若另一实例已经提交，直接返回第一组结果。唯一索引仍是跨进程最终正确性边界。

业务校验错误、StyleReview 门槛错误和故障注入错误不重试。

## HTTP 201 / 200 语义

Next.js Route Handler 使用临时 SQLite 连续调用 5 次：

| 调用 | HTTP | idempotent | 数据变化 |
|---|---:|---|---|
| 第 1 次 | 201 | false | 创建一组批准产物 |
| 第 2-5 次 | 200 | true | 无新增 |

五次返回的 `approvalRevisionId`、`voiceSampleId`、`sourceRevisionId` 一致；approvedAt、Revision 数量和 VoiceSample 数量均保持第一次批准后的值。

## 编辑后再次批准

临时库验证流程：

1. 源 Revision A 首次批准，Draft 变为 approved。
2. A 重放返回原批准产物。
3. 创建 `human_edit` Revision B，Draft 回到 editing，旧 approvedAt 清空。
4. 批准 B，创建第二组批准 Revision 和 VoiceSample。
5. B 重放返回第二组原产物。

A、B 两组的 `sourceRevisionId`、`approvalRevisionId`、`voiceSampleId` 分别不同；最终恰好两组，没有重复。

另有 `ai_suggestion` 源 Revision 测试：源本身的 `approvedSourceRevisionId` 保持 NULL，批准 Revision 和 VoiceSample 都指向该 suggestion ID，而不是批准 Revision 自身。

## 事务回滚验证

使用 SQLite trigger 分别在四个位置强制失败：

| 故障点 | 结果 |
|---|---|
| StyleReview INSERT | 全部回滚 |
| human_approval Revision INSERT | StyleReview 同时回滚 |
| EditorialDraft approved UPDATE | StyleReview 和 Revision 同时回滚 |
| approved_draft VoiceSample INSERT | StyleReview、Revision、Draft 更新同时回滚 |

四种情况均确认：

- 没有孤立 StyleReview。
- 没有孤立 `human_approval` Revision。
- 没有孤立 VoiceSample。
- approvedAt 未部分更新。
- Draft status、currentRevisionId 保持原值。

## 已知限制

1. 同进程锁不是分布式锁；跨实例正确性依赖数据库唯一索引和事务。
2. 有限重试耗尽后会返回明确错误，不会无限等待；数据库唯一约束仍保证不重复写。
3. `approvedSourceRevisionId` 和 `sourceRevisionId` 是标量幂等键，没有 Prisma 外键，以避免 SQLite 为真实表做重建迁移。
4. migration 只回填迁移时“已批准且当前 Revision 为 human_approval”的合法历史记录；更早的异常孤立数据需要单独审计。
5. 本机历史上标准 `prisma migrate dev/deploy` 在 schema engine 阶段返回空错误。本次使用 `prisma db execute` 验证真实库副本，并在全新 SQLite 顺序执行完整 migration SQL，不声称标准 migrate deploy 已通过。
