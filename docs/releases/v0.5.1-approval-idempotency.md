# v0.5.1 Approval Idempotency & Data Integrity Guard

日期：2026-07-13

## 审查结论

Phase 5.1 完成。批准操作现在以“被批准的源 DraftRevision”为幂等单位；同一源 Revision 重复或并发批准只会形成一组 `human_approval` Revision 和 `approved_draft` VoiceSample。

本阶段只修改批准流程、数据模型、迁移、测试和项目文档。没有重新批准真实稿件，没有修改现有 7 条 VoiceSample 的正文、标题、评分或时间，没有进入自动发布。

## 原问题与竞态风险

修复前，`approveEditorialDraft` 按以下顺序独立写数据库：

1. 创建 StyleReview。
2. 查询当前 EditorialDraft。
3. 查询最大 revisionNumber，再创建 `human_approval` Revision。
4. 更新 EditorialDraft 的当前 Revision、状态和 approvedAt。
5. 创建 `approved_draft` VoiceSample。

这些操作不在同一事务内，也没有源 Revision 幂等键。重复调用会把刚创建的批准 Revision 当成新当前版本，再生成另一条批准 Revision 和 VoiceSample；两个并发调用还会同时读取相同的最大 revisionNumber，在写入时冲突。最后一步失败时，前面的 StyleReview、Revision 和 approvedAt 仍会遗留。

## 数据模型与幂等键

- `DraftRevision.approvedSourceRevisionId String? @unique`：只由 `human_approval` Revision 写入，值为被批准的源 Revision ID。
- `VoiceSample.sourceRevisionId String? @unique`：只由 `approved_draft` VoiceSample 写入，值为同一个源 Revision ID。

两个字段保持可空，因此原有 `ai_initial`、`ai_suggestion`、`human_edit`、手动导入 VoiceSample 不受影响。唯一约束绑定源 Revision，不绑定 EditorialDraft：批准后新增 `human_edit` Revision 会回到 `editing`，该新 Revision 可以生成新的一组批准产物；旧源 Revision 不能重复生成。

## 服务与 API 行为

- 服务先从当前 Revision 解析源 Revision。当前已是批准 Revision 时，使用其 `approvedSourceRevisionId` 回到原源 Revision。
- 命中已有批准产物时不再运行 StyleReview、不更新 approvedAt、不创建记录，直接返回第一次生成的 Revision 和 VoiceSample ID，并返回 `idempotent: true`。
- 首次批准返回 `idempotent: false`；批准 API 使用 HTTP `201`。
- 幂等重放使用 HTTP `200`，不会返回误导性的“再次创建成功”。
- 同一 Node.js 进程内，同一 EditorialDraft 的批准调用由服务层锁串行执行。
- 数据库两个唯一索引阻止绕过服务层或跨调用生成重复批准产物。

## 事务边界

首次批准的以下操作位于同一个 Prisma 交互式事务：

1. 读取 EditorialDraft、当前 Revision 和 VoiceProfile。
2. 查询源 Revision 是否已有批准产物。
3. 创建并检查 StyleReview。
4. 创建带源 Revision 幂等键的 `human_approval` Revision。
5. 更新 EditorialDraft 的 currentRevisionId、status 和 approvedAt。
6. 创建带相同源 Revision 幂等键的 `approved_draft` VoiceSample。

任一步失败，全部写入回滚。测试使用 SQLite trigger 强制 VoiceSample 插入失败，确认没有孤立 Revision、StyleReview、VoiceSample，也没有部分更新 approvedAt 或 currentRevisionId。

## 已知限制

1. 服务层锁是单 Node.js 进程内保护；跨进程最终一致性由 SQLite 唯一索引保证，不是分布式锁。
2. SQLite 在极端多进程写竞争时仍可能返回数据库忙错误；唯一约束保证不会重复写，调用方可重试并命中原批准结果。
3. 两个源 Revision 字段是标量幂等键，没有增加 Prisma 外键关系，目的是让现有 SQLite 使用增量 `ADD COLUMN`，避免重建包含真实数据的表。
4. 迁移回填只处理迁移时“已批准且当前 Revision 为 human_approval”的既有记录；如果未来导入更早的异常孤立批准数据，需要单独审计。
5. 本机历史上 `prisma migrate dev/deploy` 在 schema engine 阶段曾返回空错误；本次当前库使用 `prisma db execute` 成功执行，干净库使用完整 migration SQL 顺序验证，未声称标准 migrate deploy 已通过。
