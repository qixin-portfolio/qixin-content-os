## Release Review 发现与修复

审查先补失败测试，复现以下问题：

1. 原 `packageHash` 只覆盖平台、文案和 evidenceSnapshot，没有覆盖 factBoundary、assetBrief 和初始 publishChecklist。
2. 同一包重复标记 published 会改写首次 `publishedAt`、URL 和备注。
3. 非 published 状态可以无约束回退。
4. Markdown HTTP MIME 被写成 `text/plain`。
5. `Content-Disposition` 缺少 RFC 5987 `filename*`。
6. publication service 的扩展名省略导致独立 Node 进程无法直接加载，阻碍真正的跨实例测试。

修复后发布专项为 3 个测试文件、26 项测试通过；完整工程为 20 个测试文件、85 项测试通过。

## 数据模型

- `PublicationPackage` 关联 EditorialDraft、源 Revision、human_approval Revision 和平台，保存最终文案、四类发布前结构、创建时 hash 和人工发布记录。
- `PublicationExport` 只保存 package ID、格式、下载文件名、内容 hash 和创建时间，不保存文件、二进制内容或用户本地路径。
- 复合唯一索引：`PublicationPackage_sourceRevisionId_platform_key`。
- 导出格式：`txt`、`markdown`、`json`。
- 包状态：`ready`、`exported`、`published`、`archived`。

SQLite 下四类结构继续使用稳定 JSON 字符串字段，不重建现有表。

## 幂等键和并发行为

幂等键定义为被批准的源 Revision ID + EditorialDraft 平台，不依赖正文 hash。

连续创建 5 次的临时库结果：

- 第一次 `idempotent: false`。
- 后四次 `idempotent: true`。
- 5 次 packageId、packageHash 完全一致。
- 数据库始终只有 1 个 PublicationPackage。

跨实例测试使用两个独立 Node 进程、两个独立 PrismaClient，在同一时刻创建同一包。进程不共享模块内存锁。最终：

- 两个调用都成功返回可识别结果。
- 一个首次结果，一个幂等结果。
- packageId 和 packageHash 一致。
- 最终只有 1 条记录，没有未处理的 unique constraint 500 或部分写入。

同进程锁只是减少竞争，数据库复合唯一索引和有限的 locked/busy/P2002/P2028/P2034 重试是跨实例边界。

复合索引在数据库层允许同一个 sourceRevisionId 使用不同 platform；重复的同 sourceRevisionId + platform 返回 `P2002`。业务服务更严格：平台来自 approved EditorialDraft，不能把已批准的朋友圈 Revision 任意改标为 X。正常跨平台发布必须来自各自已批准的平台 Draft；当前真实校准仍只覆盖朋友圈。

不同源 Revision 创建第二个包的测试通过。

## packageHash 语义

`packageHash` 表示发布包创建时的内容、批准链和发布前定义，覆盖：

- platform
- title
- hook
- body
- cta
- sourceRevisionId
- approvalRevisionId
- evidenceSnapshot
- factBoundary
- assetBrief
- publishChecklist 初始定义

规范化规则：

- 所有对象 key 递归按 key 排序后序列化。
- SourceItem 在创建快照前按 ID 升序排列。
- 字符串按 UTF-8 计算 SHA-256。
- 相同数据即使对象插入 key 顺序不同，hash 仍一致。

不进入创建时 hash 的后续字段：

- publishedAt、publishedUrl、publishNotes。
- PublicationExport 数量。
- 人工检查项的后续勾选状态。

人工检查更新、一次导出和后续发布状态测试均确认 packageHash 不变。当前实现只保存可变 checklist 的当前 JSON，没有单独保存初始 checklist JSON；hash 已包含初始值，但人工修改后如需离线重算，需要按已定义的初始 false 状态重建。这是已知限制。

真实包原 hash 覆盖不足，Release Review 已在仓库外备份后只回填派生 `packageHash`：

- 旧值：`ea186007fa24f788b70f9c95777c4bf33ca4c0c5207315467f0475f9258f67e7`。
- 新值：`d4ce1b84ab954996487ddb2b1e58018069d8f2bb23cbc90905e0d1c1ad89c058`。
- 实际变化列：仅 `packageHash`。
- status 仍为 exported，publishedAt、URL、备注仍为空。

## 证据快照不可变性

测试创建包后修改原 SourceItem 的标题、正文、sourceUrl 和 sourcePath，再读取旧包：evidenceSnapshotJson 和 packageHash 均不变。页面从包内历史 JSON 渲染，没有重新拼装实时 SourceItem。

快照只包含 SourceItem ID、类型、标题、安全引用和内容 hash，不包含完整私人笔记正文。安全检查确认不含：

- 私人绝对路径。
- API Key、Token、Cookie。
- DATABASE_URL 或本地数据库地址。
- SourceItem 完整 content 字段。

HTTP(S) URL 可以保留；本地引用必须为相对路径；绝对路径转换为 null。

## 事实边界

真实透明工地包：

- confirmedFacts：产品一页纸已经形成、功能模块清单已经形成、行业案例说明已经形成。
- missingEvidence：截图、后台版本记录、代码路径整理、真实项目案例。
- prohibitedClaims：已正式上线、已有客户、用户数量、收入结果。
- unverifiedClaims：是否正式上线、是否已有客户、用户数量结果、收入结果。

这些字段只显示在检查区域和结构化 Markdown/JSON，不混入 TXT 最终文案。PATCH schema 只接受检查项和状态操作；测试额外传入正文、evidenceSnapshot 和 factBoundary 时均被忽略，历史字段及 packageHash 不变。

## 配图需求

真实 Asset 总数为 0，真实包 `existingAssetIds = []`，`missingAssets` 明确为“当前没有已确认可发布的真实项目截图”。页面显示“配图需求”和“暂无已确认可发布的真实 Asset”，没有缩略图或“已生成配图”表述。

## 检查单与状态机

自动项：approved 状态、Revision 一致、4 条证据来源、上线/客户/用户数量/收入肯定性声明检测。真实包自动项全部通过。

人工项：隐私、真实配图、图片水印和客户信息、人工预览、发布时间。真实包人工完成数仍为 0。

安全状态规则：

- `ready -> exported`：只由成功导出事务自动完成。
- `exported -> published`：必须全部自动/人工项通过且提供有效 publishedAt。
- `published -> archived`：允许，保留首次发布时间。
- `ready/exported -> archived`：允许用户放弃内容包。
- `exported -> ready`：拒绝。
- `published -> ready`：拒绝。
- `archived -> published`：拒绝。
- 重复 published：幂等返回首次记录，不改 publishedAt、URL 或备注。

测试覆盖人工项未完成、只完成部分、全部完成、URL/备注为空、API 直接发送 published、重复 published、回退和归档。所有发布测试均使用临时数据库；真实包没有执行 published。

## HTTP 下载验证

三个接口均返回 HTTP 200：

- TXT：`text/plain; charset=utf-8`。
- Markdown：`text/markdown; charset=utf-8`。
- JSON：`application/json; charset=utf-8`。

`Content-Disposition` 同时包含安全 ASCII `filename` 和 RFC 5987 `filename*=UTF-8''...`。文件名只由日期、枚举平台和经过白名单清洗的 project slug 组成；测试用 `../../透明工地\\bad:name` 得到安全名称，不含 `..`、斜杠、反斜杠或冒号。用户标题不参与文件路径。

响应头中的 Export ID、contentHash 与数据库记录一致。

## PublicationExport 语义

- 内容在事务内成功渲染后才创建记录。
- contentHash 是 HTTP 响应 UTF-8 实际字节的 SHA-256。
- fileName 与 Content-Disposition 下载名一致。
- 不保存用户下载路径、完整文件或二进制内容。
- 临时文件删除后，格式、文件名、hash 和时间记录仍有效。
- 重复导出允许多条记录；相同格式和内容 hash 一致，不同格式按各自实际字节计算。

真实数据库仍为原 3 条 Export；HTTP 审查在副本新增 3 条后删除了临时文件和副本。

## 已知限制

1. 当前真实声音和发布包验收只覆盖朋友圈，不声称 X、小红书或抖音已完成真实声音校准。
2. 事实与敏感词检测是 deterministic 规则，不能替代齐鑫的语境和隐私判断。
3. packageHash 包含初始检查单，但初始 checklist JSON 没有独立保存；人工勾选后离线重算需要按初始定义重建。
4. 同一 sourceRevisionId 的不同 platform 由数据库复合键容纳，但服务不允许绕过平台 Draft 直接改标；跨平台必须分别批准。
5. 状态是人工记录，不是平台回执。系统不能确认朋友圈是否真的发布。
6. 当前 Asset 为 0，发布前仍需人工选择并检查真实图片。
7. raw migration SQL 不是幂等脚本；重复部署依赖 Prisma migration history。本机历史上标准 migrate deploy 存在 schema engine 空错误，因此本次仍使用副本和全新 SQLite 验证 SQL。

## 为什么仍不进入自动发布

发布包通过工程审查，只能证明文本、证据、导出和人工状态边界可靠。真实包的 5 个人工检查项仍为 0，Asset 仍为 0；没有隐私检查、真实配图、平台预览和发布时间确认。朋友圈也没有可靠的公开 URL 或平台回执。

自动发布会绕过这些人工节点，并把“导出成功”误写成“真实发布成功”。因此 Phase 5.2 继续保留复制、下载和人工状态记录，不接入任何平台发布 API。
