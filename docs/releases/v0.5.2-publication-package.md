# v0.5.2 Publishable Content Package & Manual Export

日期：2026-07-13

## 产品目标

Phase 5.2 把已批准的 `EditorialDraft` 转换为可人工检查、复制和导出的发布内容包。发布包固定保存批准文本、证据快照、事实边界、配图需求和检查单。

本阶段不自动发布，不接入微信、抖音、小红书或 X 的发布 API，不自动改写 approved 文案，也没有重新批准真实稿件。

## 数据模型

新增状态：

- `PublicationPackageStatus`：`ready`、`exported`、`published`、`archived`。
- `ExportFormat`：`txt`、`markdown`、`json`。

新增模型：

- `PublicationPackage`：关联源 `EditorialDraft`、被批准的源 `DraftRevision` 和 `human_approval` Revision，保存最终文案、四类发布前数据、`packageHash` 与人工发布记录。
- `PublicationExport`：保存每次导出的格式、文件名、内容 SHA-256 和时间。

SQLite 不支持 Prisma `Json` 字段，因此四类结构以稳定 JSON 字符串保存：`evidenceSnapshotJson`、`factBoundaryJson`、`assetBriefJson`、`publishChecklistJson`。

## 发布包幂等语义

幂等键是：

```text
sourceRevisionId + platform
```

数据库唯一索引：`PublicationPackage_sourceRevisionId_platform_key`。

创建服务只接受状态为 approved、当前 Revision 为 `human_approval`、且 `approvedSourceRevisionId` 指向同 Draft 非批准源 Revision 的 EditorialDraft。服务还会核对源 Revision、批准 Revision 和 Draft 的标题、Hook、正文、CTA 完全一致。

同一源 Revision + platform 重复创建时返回已有包，第一次 API 返回 `201`，后续返回 `200` 和 `idempotent: true`。并发写入由同进程锁减少竞争，有限重试处理 SQLite busy/locked 与唯一冲突，数据库复合唯一索引是最终正确性边界。正文 hash 不承担幂等职责。

新的 `human_edit` 或 `ai_suggestion` Revision 经过独立批准后有新的 sourceRevisionId，因此可以生成新的发布包。

## 证据快照

创建时在事务内冻结：

- EditorialDraft、源 Revision、批准 Revision、MasterContent、EventCard ID。
- approval timestamp 与 package creation timestamp。
- 每个 SourceItem 的 ID、类型、标题、安全 source reference 和正文 SHA-256。

快照不保存 SourceItem 完整原文。HTTP(S) URL 和相对路径可以作为引用；私人绝对路径被置空。SourceItem 后续发生变化时，旧包中的 JSON 和 `packageHash` 不变。

`packageHash` 覆盖平台、标题、Hook、正文、CTA 和完整证据快照。

## 事实边界

透明工地真实包固定记录：

- confirmedFacts：产品一页纸、功能模块清单、行业案例说明已经形成。
- missingEvidence：截图、后台版本记录、代码路径整理、真实项目案例。
- prohibitedClaims：已正式上线、已有客户、用户数量、收入结果。
- unverifiedClaims：是否正式上线、是否已有客户、用户数量结果、收入结果。

这些结构只用于发布前检查，没有写入朋友圈最终正文。

## 配图需求

透明工地建议类型为产品页面截图、功能模块界面、资料整理过程照片。当前数据库 Asset 数量为 0，因此：

- `existingAssetIds = []`。
- `missingAssets` 明确记录当前没有已确认可发布的真实项目截图。
- `avoidElements` 禁止虚构客户截图、虚构用户数据界面和无法核验的上线或成交数据。

配图层只描述需求和隐私风险，不生成图片，不把建议写成现有资产。

## 发布检查单

自动项验证 approved 状态、Revision 文案一致、证据存在，以及没有肯定性的上线、客户、用户数量和收入声明。

人工项包括隐私检查、真实配图选择、图片水印与客户信息检查、人工预览和发布时间确认。自动项不可通过 API 冒充手动完成；人工项未全部完成时允许导出，但服务拒绝标记 `published`。

## 导出格式

- TXT：只含 Hook、正文、CTA 中非空的最终文案，不含标题、证据 ID、系统说明或检查单。
- Markdown：包含标题、平台、最终文案、配图需求、事实边界、发布检查单和证据摘要。
- JSON：包含结构化发布包、证据快照、事实边界、配图需求和检查单，不含数据库配置、私人绝对路径或完整 SourceItem 私人原文。

每次导出都会创建 `PublicationExport`。相同格式重复导出可产生新记录；内容不变时 `contentHash` 一致。接口以内存响应交付文件，不在仓库内写生成包。

## 人工发布状态

发布包创建后为 `ready`，首次导出后记为 `exported`。标记 `published` 前必须满足：

1. Draft 仍指向同一 approved Revision。
2. 自动检查全部通过。
3. 人工检查全部勾选。
4. 提供有效 `publishedAt`。

`publishedUrl` 和 `publishNotes` 可为空。系统只保存用户的人工记录，不检查朋友圈是否真实发布，也不调用平台。

## 页面与 API

- `/publication`：发布包台账、状态、导出次数、检查进度，以及 approved Draft 创建入口。
- `/publication/[packageId]`：三栏证据台账；最终文案只读，可复制和导出；同时展示证据、事实边界、配图需求、自动/人工检查和人工状态。
- `GET/POST /api/publication`：列表和幂等创建。
- `GET/PATCH /api/publication/[packageId]`：详情、人工检查单和状态更新。
- `POST /api/publication/[packageId]/export`：TXT、Markdown、JSON 导出。

详情页明确显示：“该文案来自已批准 Revision，修改需重新进入编辑流程。”

## 已知限制

1. 只有朋友圈有 7 条真实 VoiceSample 和当前真实验收；模型虽保持平台通用，不声称其他平台已完成声音校准。
2. 事实边界和禁止声明检测是 deterministic 规则，不能替代齐鑫对语境、隐私和事实的人工判断。
3. evidence snapshot 保留哈希而非 SourceItem 全文，历史原文需要回到受控素材库核对。
4. 发布状态是人工记录，不是平台回执；系统无法确认用户是否真的发布。
5. 同进程锁不是分布式锁，跨实例最终正确性依赖 SQLite 唯一索引和有限重试。
6. 当前没有真实可发布 Asset，发布前仍需人工选择并检查图片。
7. 本机 Prisma 7.8 历史上标准 `migrate dev/deploy` 在 schema engine 阶段返回空错误；本次使用 `prisma db execute` 验证真实库和副本，并在全新 SQLite 顺序执行全部 migration SQL，不声称标准 migrate deploy 已通过。

## 为什么不做自动发布

批准文案不等于已完成隐私检查、配图选择、平台预览和发布时间判断。朋友圈还没有稳定公开 URL 或可靠发布回执，当前也只有朋友圈完成首批真实声音校准。自动发布会绕过本阶段刻意保留的人工事实与隐私节点，因此 Phase 5.2 只提供复制、导出和人工状态记录。
