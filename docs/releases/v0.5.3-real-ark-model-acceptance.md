## Previous Direct-Expansion Result

The initial direct-expansion acceptance was rejected for content quality. With sparse inputs, the Character model added unsupported time, place, action, object, and bodily-feeling details. It is not acceptable to treat a fluent expansion as a factual user record.

## Phase 5.3.4 Follow-up

Drafting now collects optional user details first and requires every concrete claim to cite a literal quote from the raw input or a supplied answer. This prevents unsupported text from being offered as a candidate; it does not make the model intrinsically factual.

## Phase 5.3.5 Revalidation

The literal `sourceQuote` contract was too strict for valid rewrites, so it has been retired. Each drafts request now creates a request-local FactLedger from only the raw input and non-empty user detail answers. The model cites stable IDs (`F1`, `F2`, and so on), and the server rejects empty or unknown IDs, unsupported concrete details, and external claims that lose attribution. FactLedger entries, IDs, and interpretation metadata do not enter the API candidate payload, browser UI, local session, database, or Git.

The same candidate model was revalidated without fallback. This is a guardrail revalidation, not a claim that the model is ready to be the default content writer.
