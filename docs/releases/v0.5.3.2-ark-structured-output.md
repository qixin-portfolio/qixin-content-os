## 2. 结构化输出方案

当前实际方案为 `response_format: { "type": "json_object" }`，没有使用 `json_schema`。原因是当前模型和 Chat API 的 `json_schema` 能力没有可靠实测证据，本轮不猜测请求参数。

解析和校验顺序：

1. 直接 `JSON.parse`。
2. 如果完整内容只被一层 Markdown JSON 围栏包裹，移除围栏后再解析。
3. 含围栏外解释文字或任意前后缀时，判定 `schema_validation_failed`。
4. 解析结果通过严格 Zod Schema；不使用类型断言跳过校验。
5. 首次失败允许一次结构修复；修复只能补空值或纠正 JSON 类型，不能新增事实、经历、情绪、判断、结果或下一步。
6. 第二次仍失败时返回 `schema_validation_failed`，不自动 fallback。

## 3. Schema 与事实边界

topics 由一次模型请求返回统一 `TopicGenerationEnvelope`：一个 brief 和正好 3 条 topics。浏览器传回的 brief 不能覆盖服务端结果；服务端再次按原始输入过滤 confirmedFacts、具体细节、下一步和来源依据。

允许的规范化：

- `personalReaction`、`personalJudgment`、`unresolvedQuestion`、`possibleNextStep` 的 null/缺失值转为空字符串。
- 数组字段的 null/缺失值转为空数组。
- 数组字段收到单字符串时，转为单元素数组。
- 字符串去除首尾空格，并执行字段长度限制。

禁止的规范化：

- 不生成缺失事实、个人感受、判断、结果或下一步。
- 不用固定模板补选题或正文。
- 不从长文本模糊截取局部 JSON 冒充成功。
- 不接受额外字段绕过边界。

drafts 初次调用一次返回正好三种：`scene_record`、`thought_progression`、`restrained_short`。服务端继续执行事实边界、结构相似度、VoiceSample 整句复制和禁用模板短语检查；只对质量失败版本定向重试一轮。

## 4. VoiceSample 与 Prompt 边界

- 结构画像只使用全部 5 分样本和最多两条 4 分样本；3 分样本不进入画像输入。
- 不向 Ark 发送 VoiceSample 内部标题或完整正文。
- Provider 只收到聚合后的段落、开头、判断位置、留白、不确定性、自嘲和情绪强度等结构特征。
- 请求与错误日志不输出 Prompt、API Key、VoiceSample 正文或完整模型响应。

## 5. Timeout 与 fallback

- Ark Provider 超时：120 秒。
- `/api/create/topics` 与 `/api/create/drafts` Route 上限：150 秒。
- topics 超过 25 秒、drafts 超过 35 秒标记 `slowResponse=true`，但未超时时不自动判失败。
- timeout、authentication_failed、model_or_endpoint_not_found、rate_limited、schema_validation_failed 均返回明确错误，不自动 fallback。
- `ARK_API_KEY` 或 `ARK_MODEL_ID` 缺失时返回明确配置错误，也不静默 fallback。
- 只有用户主动点击“使用本地演示生成”后，Route 才直接选择 `LocalFallbackProvider`，不会先浪费一次 Ark 调用。
- 页面提示固定为：“本地演示内容可能带有模板感，不代表真实模型效果。”
- 失败不会清除 localStorage 中的原始输入或人工编辑。

## 6. 模型调用次数

正常路径的最少调用次数：

- brief + topics：1 次。
- drafts 三稿：1 次。
- 首次结构不合格：对应步骤最多增加 1 次结构修复。
- 三稿质量检查不通过：只对命中的版本定向重试一轮。

不存在 ContentBrief 一次、TopicCandidates 再一次的固定双请求，也不存在初次三稿分别调用三次。
