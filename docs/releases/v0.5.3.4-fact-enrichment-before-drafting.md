# Phase 5.3.4 Fact Enrichment Before Drafting

> Historical note: the literal `sourceQuote` field described below was replaced in Phase 5.3.5 by a request-local FactLedger and valid `factIds`. The product flow, optional detail step, sparse mode, one directed repair, and partial-success behavior remain in force.

## Product Flow

`raw input -> topics -> optional fact questions -> drafts -> human edit -> copy`

The detail step is local-only. It asks at most three input-specific questions, accepts blank answers, and offers sparse mode: "没有更多细节，直接写短一点". `factQuestions`, `factAnswers`, and `detailMode` are stored only in the browser's local create session.

## Fact Contract

The drafts provider receives only the raw input, selected topic, non-empty user answers, source mode, and the existing compact voice summary. Concrete draft claims require `usedFacts` with a `sourceQuote` that is a literal substring of raw input or an answer. Unsupported concrete time, place, action, object, bodily sensation, project result, quotation, or number rejects that draft. Abstract `inferredStatements` may not carry concrete facts.

## Targeted Repair and Partial Success

- Initial generation still requests the three draft types together.
- Each failed type receives at most one independent repair request.
- The repair contract is limited to raw input, fact answers, detail mode, selected topic, failing type, and reasons. It excludes successful draft text, VoiceSample text, internal titles, and database identifiers.
- Passing drafts remain unchanged. A repaired pass is `repaired`; an unresolved failure is `rejected_for_ungrounded_details` and is not returned as a selectable card.
- The API can validly return one or two cards. It never uses local fallback to fill a rejected result.

## Test Coverage

The focused tests cover all requested behavior: no repair for three passes; one- and two-type repair; unchanged passing identifiers and bodies; minimal repair input; no VoiceSample/internal-ID leakage; `repaired` and rejected states; hidden rejected output; one-repair maximum; provider failure preserving passes; no fallback; local session persistence; and manual-edit isolation.

## Real Acceptance

- A sparse produced three short grounded drafts without added scene details.
- A enriched, C sparse, and C enriched produced no selectable drafts because their source quotes were not literal user quotes even after the one allowed repair. The guardrail held.
- B sparse produced three grounded drafts without invented scene details, but their structure was too similar for a full quality pass.
- B enriched is `pending_user_details`: Qixin has not supplied a real location, remembered scene, or feeling. No developer, test, or model substitute was used.

## Browser and Local Drafts

The browser flow confirms the detail screen, three-question maximum, blank answers, sparse mode, refresh recovery, and returning to topic selection without losing answers. A compatibility migration now fills the new detail fields for old local sessions instead of discarding their existing manual editor content.

During this work, an already-loaded legacy browser session first reached the pre-migration incompatibility path after a development-server restart. That session could not be recovered from the page once an empty session was saved. The compatibility test and migration prevent this path for future reloads; this incident must not be described as successful preservation of that already-replaced session.

## Boundaries and Limits

- No Prisma model, migration, database record, VoiceSample, approved draft, or publishing state changed.
- No secrets, full prompts, full VoiceSample text, or raw Ark responses are stored in Git.
- The strict literal-quote rule currently rejects some otherwise plausible model rewrites. Do not weaken it silently to improve completion rates.
- This phase is not ready to claim enriched acceptance across all content types.

## Final Review Note

The original `sourceQuote` transport is historical only; Phase 5.3.5 replaced it with FactLedger IDs before final human acceptance on 2026-07-16. The local-only optional-detail flow, sparse mode, refresh recovery, human editing, and copy flow were accepted in real `/create` use across project, life, and external-opinion inputs. B enriched remains `pending_user_details`; no placeholder detail was introduced for release review.
