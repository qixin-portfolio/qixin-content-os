# Phase 5.3.5 Fact-ID Grounding

## Purpose

Phase 5.3.4 correctly rejected unsupported detail, but its `usedFacts[].sourceQuote` contract required a model-generated quote to be a literal user substring. Valid rewrites therefore failed for the wrong reason. Phase 5.3.5 replaces that transport with server-created FactLedger IDs. It does not weaken the fact boundary.

## FactLedger Contract

For every drafts request, the server builds a temporary ledger from only:

- raw user input, split conservatively on sentence punctuation and line breaks;
- each non-empty fact-answer field as one additional entry;
- source mode and explicit external-opinion/user-judgment classification.

Each entry has a request-local ID (`F1`, `F2`, ...), `sourceType`, and conservative category. The model never creates or expands ledger facts. The ledger is not persisted, returned to the browser, placed in localStorage, or shown in the UI.

The model draft contract is:

```ts
{
  type: "original_record" | "restrained_judgment" | "minimal_expression";
  content: string;
  usedFacts: [{ claim: string; factIds: string[] }];
  interpretations: [{ text: string; basisFactIds: string[] }];
}
```

The server rejects empty or invalid IDs. An interpretation must cite its basis and cannot add concrete time, place, action, object, bodily sensation, quotation, number, or project result. An external-opinion fact can be used only when the body visibly retains outside ownership.

## Repair and UI Boundary

Initial generation still asks for all three drafts once. A failed type alone may receive one repair. The repair input contains only the FactLedger, allowed IDs, selected topic, detail mode, failed type, and rejection reasons. It excludes passing candidate text, VoiceSample text, internal titles, database IDs, and fallback data.

Passing candidates remain unchanged. A successful repair is `repaired`; a second failure is `rejected_for_ungrounded_details` and is excluded from selectable cards. The API can return one or two cards. `usedFacts` and `interpretations` are stripped before candidates reach the frontend, and old saved sessions remove those retired fields while preserving manual editor content.

## Quality Result

The new contract fixes false rejection of ordinary rewrites and continues to block unsupported detail. Sparse output is shorter and avoids fabricated scenes, but the model often differentiates the three versions by compression or word order rather than genuinely distinct structure. This phase validates grounding behavior, not default-model writing quality. B enriched remains `pending_user_details`.

## Browser and Data Boundary

The existing `/create` browser session showed the optional detail step, at most three questions, populated answers, and no visible FactLedger IDs. The isolated browser-control tab returned an unstable visual-control response before it could complete a non-destructive end-to-end replay; this is not counted as a full browser interaction pass. Unit tests cover fact-answer refresh recovery, returning to topics, rejected-card omission, repaired candidate availability, manual-edit isolation, and legacy session migration.

No Prisma schema, migration, database content, VoiceSample, approved draft, or publication state changed. No automatic publishing, merge, or push was performed.

## Known Limits

- The local high-risk detail detector is conservative and lexical; Fact IDs are the primary contract, not a semantic proof of every sentence.
- A valid Fact ID citation does not itself make the writing useful or structurally varied.
- Do not infer B enriched facts. Wait for Qixin's actual detail input.
