# `/create` 模板化生成审计

日期：2026-07-14
范围：`POST /api/create/topics`、`POST /api/create/drafts`、create generator、StyleReview 和对应测试
结论：Phase 5.3 的三稿并非内容生成，而是把用户输入插入少量固定句子。真实验收不通过成立。

## 1. 当前调用链

```text
用户输入
  -> generateCreateTopics
  -> 选择 topic
  -> generateCreateDrafts
  -> contentOsDrafts 或 genericDrafts
  -> StyleReview 只生成提示
  -> 返回三稿
```

两条 API 都没有调用真实模型。选题和正文全部由 TypeScript 字符串模板同步生成。

## 2. 选题模板

### Content OS 特例

输入同时命中 `Content OS|内容系统` 和 `功能.*多|真正需要|每天.*写` 时，固定返回：

1. “功能越来越多，为什么反而更难开始写？”
2. “我真正需要的，不是更多功能”
3. “做内容系统时，我差点忘了最初的问题”

标题、价值说明和切入角度全部硬编码。用户原始输入只负责触发分支，不参与焦点提取。

### 通用路径

除特例外，固定返回：

1. `先把“{输入前 28 字}”这件事记下来`
2. “这件事改变了我什么判断？”
3. “现在能确认的，和还没做完的”

这里没有识别事件、人物、外部观点、未上线状态、生活场景或真实矛盾。第二、第三条可以套用到任何输入，因此三条只是通用写作姿势，不是三个内容焦点。

## 3. 草稿中的硬编码模板

### 3.1 开头模板

- A 通常直接完整复述用户输入。
- B 的 Content OS 特例固定以“最近做 Content OS，有一个很直接的感受”开头。
- B 通用稿仍先完整复述用户输入，再插入“我现在更在意的是”。
- C 始终完整复述用户输入。

问题：A/C 首段相同；B 只是把固定判断放到输入前面，没有形成独立叙事逻辑。

### 3.2 段落模板

Content OS 特例固定为三段：

1. 输入复述或固定判断。
2. “功能变多不等于问题解决”的抽象解释。
3. “回到每天知道写什么”的下一步。

通用路径固定为一至三段：

1. 完整复述输入。
2. 解释“事实和判断分开”或插入选题标题。
3. 用“先记到这里”结束。

问题：所谓克制短版只是删短 A/B 的解释，没有改变信息组织方式。

### 3.3 转折模板

硬编码转折包括：

- “做到这里才发现”
- “并不是一回事”
- “现在还没有完整答案”
- “功能当然有用，但如果……”
- “我现在更在意的是”
- “先不急着把它说成一个确定结论”
- “能确认的先写，不能确认的先留着”

这些转折不来自 ContentBrief，也没有确认用户是否真的产生了对应感受或判断。

### 3.4 结尾模板

硬编码结尾包括：

- “我先把这个偏差记下来”
- “接下来想先把……做顺，其他功能再慢慢看”
- “这个判断还要放回每天的使用里继续验证”
- “先回到……，其他的以后再加”
- “后面再慢慢分清”
- “能确认的先写，不能确认的先留着”
- “先记到这里。还没做完，也不用急着把话说满”

问题：即使输入没有下一步、没有“偏差”或没有未完成判断，系统仍自动补一个克制式收束。

### 3.5 三稿共用句式和结构

三稿共同依赖：

- 完整复述用户输入。
- “不等于 / 并不是”式抽象判断。
- “还没有完整答案 / 还没做完”式不确定性。
- “先……，其他……”式结尾。
- “记录、确认、判断、验证”这组系统编辑词。

它们共同形成：

```text
输入复述 -> 抽象结论 -> 未完成或下一步收束
```

因此 A、B、C 不是三种叙事逻辑，只是同一结构的顺序调整和长度变化。

## 4. VoiceSample 当前实际参与方式

当前 7 条 `wechat_moments` VoiceSample 的处理方式：

1. 过滤 active、approved、朋友圈样本。
2. 用 `qualityRating` 和 `sourceType` 计算权重后排序。
3. 把排序后的样本传给旧 StyleReview。
4. StyleReview 将 `sample.title + sample.body` 拆词，计算关键词重合形式的 authenticityScore。
5. 草稿正文生成完全不读取上述结构信号；排序也不改变最终正文。

实际问题：

- `approved_draft` 和高分样本虽然排序靠前，但旧匹配对所有样本仍等权。
- Content OS 内部索引标题被当成声音文本参与匹配。
- 没有提取开头方式、观点位置、段落节奏、留白、不确定性或情绪强度。
- authenticityScore 不在 `/create` 展示，也不影响三稿生成。
- 因此当前“参考 VoiceSample”只存在于名义调用链中，没有实际塑造草稿。

## 5. 测试为什么没有拦住模板问题

原测试主要验证：

- 返回三个固定名称。
- 文本包含用户输入中的 `Content OS`。
- 不复制一条指定 VoiceSample 完整句子。
- 权重函数数值顺序正确。
- 提示数量不超过三条。

缺失的行为测试：首句差异、结尾差异、段落结构差异、外部观点归属、未上线边界、生活内容不升华、无下一步不补下一步、相似度失败与重试。

所以测试证明了接口结构稳定和基本事实词存在，没有证明内容不是模板。

## 6. Provider 审计

- 项目现有 `src/lib/ai/content-generator.ts` 是 deterministic mock，不是可调用的 Provider abstraction。
- 审计开始时没有可供 `/create` 复用的真实模型客户端，也没有 AI SDK 依赖。
- Phase 5.3.1 产品决策只接火山方舟 Seed 2.1，不接 Grok、Qwen 或 DeepSeek。
- 新 Provider 名称固定为 `volcengine_ark`，只从服务端读取 `ARK_API_KEY` 与 `ARK_MODEL_ID`。
- `ARK_MODEL_ID` 必须由齐鑫从火山方舟控制台提供真实模型 ID 或推理接入点 ID，代码不猜测、不硬编码 `doubao-2.1`。
- 当前本地没有配置上述两个变量，本轮没有发出真实模型请求。

最小调整方向：建立单一服务端 Provider interface，由 Route 通过 factory 取得 `volcengine_ark` 或 `deterministic_fallback`。Ark 使用官方 Chat API 的 JSON 输出，并继续通过 Zod 校验；没有配置或调用失败时明确降级，不能把 fallback 结果标记成 Seed 2.1 或个性化真实模型生成。

## 7. 修复边界

修复不能继续增加更多完整文案模板。需要改为：

1. 先生成可校验的 ContentBrief，只保留输入中存在的信息。
2. 再按场景记录、思考推进、克制短版三种不同叙事计划生成。
3. VoiceSample 只提取加权结构特征，内部标题不参与。
4. 三稿生成后执行结构相似度检查；只重试重复版本一次。
5. 真实模型不可用时显示本地演示模式，不声称已个性化生成。
6. 用户人工编辑稿继续独立保存在 localStorage，任何重试都不能覆盖。
